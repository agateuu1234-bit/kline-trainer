# Plan 1f — M0.1 Schema Versioning Contract 设计 Spec

**Status**: drafted 2026-04-22；本文件是 Plan 1f 的设计 spec（brainstorming 输出）；后续走 writing-plans → subagent-driven-development → codex adversarial review → PR。

## 1. 目标

把 `kline_trainer_modules_v1.4.md` M0.1 章节中与 schema 版本管理相关的三类规则冻结为 repo 内权威治理 doc：

1. **CONTRACT_VERSION 矩阵**（spec L133-148）：顶层 `"1.4"` + 5 个子版本维度（PostgreSQL / 训练组 SQLite / app.sqlite / Swift 模型 / P2 journal states）+ 各自变更触发 bump 的条件
2. **Bump 策略二分**（spec L137-138 + L287-291）：破坏性变更 vs 本地兼容新增 + DAO/reader 对未知 state 的处理规则
3. **Migration Rollback 规则**（spec L151-157）：PostgreSQL / app.sqlite / 训练组 SQLite 三条 owner + forward/rollback 文件对的要求

## 2. 非目标（out-of-scope）

- **不实施具体 migration SQL**（spec L161-291 的 `content_hash` 两阶段 + `download_acceptance_journal` + `0003_v1.4_purge_leased` 由 Plan 2 B3 owner 落地）
- **不定义 `CONTRACT_VERSION` 常量代码**（Python / Swift 代码层的常量定义由 Plan 2 / Plan 3 模块 PR 落地）
- **不实施 L2232 CI assert**（需先有常量存在；登记为未来强制点）
- **不动 spec**（`kline_trainer_modules_v1.4.md` 只读；发现 spec 本身 drift 时 pushback 到 Plan 1e 教训处理）
- **不吸收 M0.5 consolidation**（已 scope 为独立 Plan 1g；Plan 1f 只管 M0.1 schema 版本）

## 3. 架构

**单一治理 doc**：`docs/governance/m01-schema-versioning-contract.md`，与 `m04-apperror-translation-gate.md` / （未来）`m05-*` 同目录，parallel 结构。doc-only，零代码，零 CI workflow 新建。

**配套聚合 acceptance 脚本**：`scripts/acceptance/plan_1f_m0_1_schema_versioning.sh`。**关键设计**（吸收 Plan 1e R7 MEDIUM 教训）：每个必须 anchor 做**独立**断言，**不用聚合阈值**；删任何单一 anchor 必须能独立 FAIL。
