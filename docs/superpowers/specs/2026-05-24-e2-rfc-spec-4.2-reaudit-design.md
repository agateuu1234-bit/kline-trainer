## 一、背景：为什么需要这个 RFC

`PositionManager`（持仓管理，加权平均成本法）在 spec `kline_trainer_plan_v1.5.md` §4.2（L630-662）被定义为一个**裸 `mutating`、无错误类型的值类型**——它只做算术，假设调用方传入的都是合法数据（非法操作如空仓卖出在 UI 层被灰置按钮拦截，L664-668）。

这个"信任调用方"的设计与代码评审器（codex）的 **defense-in-depth 偏见**长期冲突：codex 按通用工程标准判断攻击面，**不**依据 spec 文本，会反复要求到处加防御性守门。实证：

- **PR #36**（PositionManager 实现）卡 codex **8 轮** needs-attention 后关闭（违反 ≥3 轮 abort 硬规则）。即使 v0 提交 `35323b2` 就是 spec 字面实现，codex R1 仍 push precondition。
- E2 brainstorming **v1 + v2 + v3 三连 abort**，被认定为"spec §4.2 设计 vs codex bias 的**模块级永久冲突**"（memory `feedback_module_level_abort_signal`）。
- 结论：E2 进 backlog，等一个**脱离 PR 上下文的独立 spec §4.2 重审窗口**——**本 RFC 就是那个窗口**。

**当前 main 实际状态（已 grep 验证）**：
- `plan_v1.5.md` §4.2 = 仅原始裸值类型设计，**无任何 rationale block**（全文只有两处"需求 4.2.1"，是手势引用，无关）。
- `modules_v1.4.md` §E2（L1489-1491）= 仅一句"见 v1.5 §4.2；类型加 Equatable"。
- main 上**无** `PositionManager.swift`（PR #36 实现从未合并）。
- v1 RFC 草稿只存在于归档分支 `spec-redesign-pr1-e2`（标记 ABORTED），从未 merge。

**v1 为什么被中止（不是技术方向错）**：第三轮对抗性评审抓出 ① 文档归属错配（M0.1/M0.3/M0.4/CONTRACT_VERSION 错挂 plan，真实在 modules）② scope 打包违规（12 子项硬塞 3 sub-task）③ 治理循环依赖（spec authority vs m01 reference 同 PR 自循环）④ 文件名/semver 命名碰撞。**技术方向（B）本身收敛且可复用。**

## 二、决议：方向 B（用户 2026-05-24 认可）

**精准防御 + 把信任边界论证写进 spec。** 一句话：把"哪里该防、哪里不该防"的论证以权威 rationale block 写进 §4.2，让 codex 无法再无止境要求加防御。

### 2.1 绑定决定（5 层防御按"入口"重新归类——保留全部 5 层，不删）

PositionManager 只有两条入口（ingress）：

```
入口 1（进程内 mutation）：
  UI → E3 TradeCalculator.quoteBuy/quoteSell → Result<Quote, TradeReason>
       ├─ .failure → UI Toast，链路终止
       └─ .success(quote) → E5 TrainingEngine → position.buy/sell
  ⟹ 传到 buy/sell 的输入【已被 E3 验证】；违约 ⟺ 调用方 programmer error

入口 2（持久化 load）：
  SQLite position_data (Data) → JSONDecoder → PositionManager.init(from:)
       └─ DecodingError throws / invariantsHold == false → load 失败，链路终止
  ⟹ 唯一真实的【外部不可信入口】（存档可能损坏/被篡改）
```

| 原 5 层（PR #36 R1-R5 收敛） | 归到哪个入口 | 处置 |
|---|---|---|
| ① precondition | 入口 1（进程内） | **trap**（程序员错误，同 Swift `Array` 越界） |
| ② 自定义 throwing decoder | 入口 2（持久化） | **throws** `DecodingError` |
| ③ invariantsHold helper | 两入口共用 | decoder 守门 + debug assert 兜底 |
| ④ overflow / isFinite 预检 | 入口 1（进程内） | mutation 前预检 → trap |
| ⑤ candidate-state guard | 入口 1（进程内） | 算出候选状态→校验→提交（buy/sell 的 mutation 模式） |

**核心**：5 层全保留，但 **2 层（②③）守真实的磁盘外部边界用 throws；3 层（①④⑤）防进程内 programmer error 用 trap**。rationale block §4.2.3 显式论证"进程内二次 throws 守门 ROI 为负"——这就是钉死 codex 攻击面的机制：它无法把进程内的 trap 升级成"必须 throws"，因为 spec 已论证为什么不该。

**buy/sell 不改 throws、不改 Result。** 理由见 §4.2.5 否决方案表。

## 四、§4.2 rationale block 大纲（§4.2.1–§4.2.8）

复用 v1 归档草稿已收敛文本（见附录 A），**唯一实质性适配 = bump 决策推迟**（见第五节）。

- **§4.2.1 Trust-boundary 定位** — 两入口图（同 2.1）；contract 不绑定具体 caller 身份。**【适配】** v1 此处的"Bump 决策（M0.3 1.3→1.4 / CONTRACT_VERSION 1.4→1.5）"段落 → 改写为"bump 判定推迟顺位 8"（第五节）。
- **§4.2.2 Swift stdlib 处置一致性** — programmer error/contract violation → trap（`Array` 越界、`Int.+`）；I/O/parse/external-input → throws（`JSONDecoder`、`String(contentsOf:)`）。
- **§4.2.3 Threat model** — 进程内输入域 vs 类型上界 gap 表（见附录 A，量级估算，不作 contract 上界）；论证进程内二次守门 ROI 为负。
- **§4.2.4 数值溢出语义** — Int 路径 trap on overflow；Double 路径 IEEE 754 saturating；mutation 前 `addingReportingOverflow` / `isFinite` 预检 → 违约 precondition trap。
- **§4.2.5 Considered alternatives** — buy/sell throws ❌ / `Result<_,_>` 与 E3 统一 ❌（不同 error class，false symmetry）/ 进程内 defense-in-depth ❌（ROI 为负）。
- **§4.2.6 Acceptance 与 testability** — precondition = fail-fast，不作正常控制流；buy/sell 违约路径**不在 test surface**（contract 假设上游已守门）；trap message 含违约参数 + 当前 state。
- **§4.2.7 Migration note** — 假设 v1 sandbox 单用户、无外部 import；v2 若引入云同步/跨设备/CSV import 等新 ingress，§4.2 contract 须重评（可能升 throws/Result）。
- **§4.2.8 invariantsHold contract**（v1 §M0.4.X 内联至此）— `private static func invariantsHold(shares:averageCost:totalInvested:) -> Bool`；O(1)；校验 `shares ≥ 0` ∧ `totalInvested ≥ 0` ∧ `totalInvested.isFinite` ∧ `(shares==0)⟺(totalInvested==0)` ∧ `shares>0 ⟹ averageCost>0 ∧ averageCost.isFinite ∧ averageCost*shares ≈ totalInvested`；使用点 = decoder（false→throw）/ buy-sell 候选态（false→trap）/ debug assert。

## 七、停止规则（永久偏见护栏）

本 RFC 自身也要过 opus 4.7 xhigh 对抗性评审。规则：

- **§4.2 rationale block 一旦写入即为权威答案。** 评审若在论证已给出后，仍反复纠缠"信任边界决定本身"（要求 buy/sell 改 throws、要求进程内加二次守门），判定为 **permanent-bias 复述** → 按 memory `feedback_codex_round6_self_contradiction` 拒绝，走 user TTY override + admin merge，**不**实施。
- **新 finding（非复述）仍照常修。** 例如文档归属、内部矛盾、引用悬空、量级表算错——这些是真 finding，修。
- 区分标准：finding 是否针对"已被 §4.2.3/§4.2.5 显式论证并否决的点"。是 → 复述；否 → 真 finding。

**§4.2.3 Threat model 量级表（v1 round-1 已修正版）**

| 量 | 真实上界（量级） | 类型上界（量级） | gap |
|---|---|---|---|
| `averageCost`（Double, 元/股） | ~10⁴ | ~10³⁰⁸ | ~304 |
| `shares`（Int, 股） | ~10⁶–10⁷ | ~10¹⁸–10¹⁹ | ~12–13 |
| `totalInvested`（Double, 元） | ~10¹⁰ | ~10³⁰⁸ | ~298 |
