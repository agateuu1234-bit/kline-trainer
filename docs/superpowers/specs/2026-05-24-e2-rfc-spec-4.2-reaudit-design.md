## 一、背景：为什么需要这个 RFC

这个"信任调用方"的设计与代码评审器（codex 或 opus 对抗性评审）的 **defense-in-depth 偏见**长期冲突：评审按通用工程标准判断攻击面，**不**依据 spec 文本，会反复要求到处加防御性守门。实证：

- 结论：E2 进 backlog，等一个**脱离 PR 上下文的独立 spec §4.2 重审窗口**——**本 RFC 就是那个窗口**。

**当前 main 实际状态（已 grep 验证）**：

## 二、决议：方向 B（用户 2026-05-24 认可）

**精准防御 + 把信任边界论证写进 spec。** 一句话：把"哪里该防、哪里不该防"的论证以权威 rationale block 写进 §4.2，让评审无法再无止境要求加防御。

### 2.1 绑定决定（5 层防御按"入口"重新归类——保留全部 5 层，不删）

PositionManager 有以下入口（ingress）——R1-High#5 补全 force-close 与 replay 两条进程内入口：

```
入口 1a（进程内 mutation，交易路径）：
  UI → E3 TradeCalculator.quoteBuy/quoteSell → Result<Quote, TradeReason>
       ├─ .failure → UI Toast，链路终止
       └─ .success(quote) → E5 TrainingEngine → position.buy/sell
  ⟹ 此路径输入【已被 E3 Result 通道验证】；违约 ⟺ 调用方 programmer error

入口 1b（进程内 mutation，局终强平路径）：
  E5 局终（globalTickIndex >= maxTick）→ E3 TradeCalculator.forceCloseOnEnd(...) -> SellQuote（裸报价，无 Result）
       → position.sell(shares:)
  ⟹ forceCloseOnEnd 按构造产出 shares == holding（≤ 当前持仓）或非法输入时全零报价（sell(0) 为 no-op）；
     precondition 容许 sell(0)；此路径不经 Result 通道，但按构造不可越界（见 §4.2.1 显式论证）

入口 1c（进程内 reconstruction，replay/review）：
  E6 TrainingSessionCoordinator.replay/review → 重建运行时
  ⟹ 若经持久化载入 → 归入口 2（decoder 守门）；若经重放 trade ops → 每步仍走入口 1a/1b

入口 2（持久化 load）：
  SQLite position_data (TEXT base64) → Data → JSONDecoder → PositionManager.init(from:)
       └─ DecodingError throws / invariantsHold == false → load 失败，链路终止
  ⟹ 唯一真实的【外部不可信入口】（存档可能损坏/被篡改）
```

| 原 5 层（PR #36 R1-R5 收敛） | 归到哪个入口 | 处置 |
|---|---|---|
| ① precondition | 入口 1a/1b/1c（进程内） | **trap**（程序员错误，同 Swift `Array` 越界） |
| ② 自定义 throwing decoder | 入口 2（持久化） | **throws** `DecodingError` |
| ③ invariantsHold helper | 两入口共用 | decoder 守门 + debug assert 兜底 |
| ④ overflow / isFinite 预检 | 入口 1（进程内） | mutation 前预检 → trap |
| ⑤ candidate-state guard | 入口 1（进程内） | 算出候选状态→校验→提交（buy/sell 的 mutation 模式） |

**核心**：5 层全保留，但 **2 层（②③）守真实的磁盘外部边界用 throws；3 层（①④⑤）防进程内 programmer error 用 trap**。rationale block §4.2.3 显式论证"进程内二次 throws 守门 ROI 为负"——这就是钉死评审攻击面的机制：它无法把进程内的 trap 升级成"必须 throws"，因为 spec 已论证为什么不该。

**buy/sell 不改 throws、不改 Result。** 理由见 §4.2.5 否决方案表。

## 四、§4.2 rationale block 大纲（§4.2.1–§4.2.8）

复用 v1 归档草稿已收敛文本（见附录 A），适配项见第六节。

- **§4.2.1 Trust-boundary 定位** — 列全部入口（§2.1 的 1a/1b/1c/2）；**显式论证 force-close（入口 1b）按构造不可越界**（`forceCloseOnEnd` 产 shares==holding 或全零，sell(0) 为 precondition 容许的 no-op），不依赖"E3 Result 验证一切"的笼统断言（R1-High#5）；contract 不绑定具体 caller 身份。
- **§4.2.2 Swift stdlib 处置一致性** — programmer error/contract violation → trap（`Array` 越界、`Int.+`）；I/O/parse/external-input → throws（`JSONDecoder`、`String(contentsOf:)`）。
- **§4.2.3 Threat model** — 进程内输入域 vs 类型上界 gap 表（见附录 A）；**数字为粗量级工程估算，明确标注非 spec 引用**（R1-High#4：spec 内无价格区间 / 无"日内笔数"上界）；论证进程内二次守门 ROI 为负。
- **§4.2.4 数值溢出语义** — Int 路径 trap on overflow；Double 路径 IEEE 754 saturating；mutation 前 `addingReportingOverflow` / `isFinite` 预检 → 违约 precondition trap。
- **§4.2.5 Considered alternatives** — buy/sell throws ❌ / `Result<_,_>` 与 E3 统一 ❌（不同 error class，false symmetry）/ 进程内 defense-in-depth ❌（ROI 为负）。
- **§4.2.6 Acceptance 与 testability** — precondition = fail-fast，不作正常控制流；buy/sell 违约路径**不在 test surface**（contract 假设上游已守门）；trap message 含违约参数 + 当前 state。
- **§4.2.7 Migration note** — 假设 v1 sandbox 单用户、无外部 import；v2 若引入云同步/跨设备/CSV import 等新 ingress，§4.2 contract 须重评（可能升 throws/Result）。**含顺位 8 bump 绑定门（见第五节）。**
- **§4.2.8 invariantsHold contract**（v1 §M0.4.X 内联至此）— `private static func invariantsHold(shares:averageCost:totalInvested:) -> Bool`；O(1)；校验 `shares ≥ 0` ∧ `totalInvested ≥ 0` ∧ `totalInvested.isFinite` ∧ `(shares==0)⟺(totalInvested==0)` ∧ `shares>0 ⟹ averageCost>0 ∧ averageCost.isFinite ∧ averageCost*shares ≈ totalInvested`。**"≈" 必须用相对容差**（R1-Medium#6）：`abs(averageCost*Double(shares) - totalInvested) <= tol * max(1, abs(totalInvested))`，`tol` 取一个能吸收 "JSON Double 十进制往返 + buy 的 `newTotal/newShares` 除-乘非结合误差" 的小常数（顺位 8 定具体值并给测试）。**理由**：`buy` 存 `averageCost=newTotal/newShares`、`totalInvested=newTotal`，decoder 复检 `(newTotal/newShares)*newShares ≈ newTotal` 非位精确 → 用 `==` 或过紧 epsilon 会拒收 app 自己写出的合法存档。`sell` 安全（它 `totalInvested = averageCost*shares` 是精确赋值）。使用点 = decoder（false→throw）/ buy-sell 候选态（false→trap）/ debug assert。

## 五、bump 责任：执行推到顺位 8，但**不否认 bump 应当发生**（R1-Critical#1/#2 + High#3 修）

**先纠正 v1 偏差的纠正**：v1 R2 对抗性评审**已结论"bump 应当发生"**（auto-synth → throwing custom init 是 caller-visible 行为变化，命中 m01 §A "改既有语义"；v1 把 M0.3 1.3→1.4 + CONTRACT_VERSION 1.4→1.5 当作既定结论 codify）。**本 RFC 不推翻该结论**（per memory `feedback_reviewer_verdict_not_authorization`：不 end-run 前评审 verdict）。

**本 RFC 推迟的是"何时 / bump 到哪个值的执行"，不是"是否 owe bump"**——区分两件事：

1. **是否 owe bump**：`position_data` 是已 shipped 的 schema 列，当前当**不透明字节**往返（任何字节都接受）。顺位 8 引入第一个 typed throwing decoder = 把该列的解释从"任何字节"收紧为"必须解码成合法 PositionManager 否则拒收" → **这是对已 shipped 持久化契约的语义收紧**，命中 m01 §A → **bump 大概率 owe**（最终判定+具体版本值由顺位 8 按 m01 §Bump 策略给）。wire 容器（TEXT base64）不变，仅 reject-set 收紧。
2. **何时执行**：bump 要改 `CONTRACT_VERSION`（`Models.swift:7` 的代码常量）+ m01 矩阵 + acceptance 脚本——这些**必须与真正的 decoder 代码同 PR 落地**。在本 docs-only RFC 里执行会①重演 v1 的"spec §4.2 改动 ↔ m01 bump trigger 同 PR 自循环"死因，②在没有 decoder 代码时改版本常量是空头。

| 推迟项 | 推迟到哪 | 约束 |
|---|---|---|
| `CONTRACT_VERSION` / m01 矩阵 / acceptance 脚本 bump | 顺位 8 | **MANDATORY，非可选**（R1-High#3）：顺位 8 必须在引入 typed decoder 的同 PR 执行 bump + m01 矩阵 + acceptance 脚本同步；不允许"判定不 bump"绕过 |
| 任何"用 typed PositionManager 读 `position_data`"的代码 | 顺位 8（required predecessor） | 顺位 8 是**前置**：顺位 9-19 任何 PR 不得在 bump 之前引入 typed `position_data` reader（避免 spec 描述的 reject 契约无版本号锚定的窗口，R1-High#3） |
| `positionTier` 占位符（plan L656 `var positionTier: Int { 0 }`） | 顺位 8 | 本 RFC 在代码块**标注目标契约**（R1-Medium#8）：档位需"初始资金"，不在 PositionManager 状态内 → 判定为 **caller-derived（E4/E5 依初始资金+当前持仓推导）**；顺位 8 **从 PositionManager 移除此 member**。本 RFC 不重写代码块结构，仅 codify 契约让顺位 8 无歧义。 |
| `PositionManager.swift` + 测试 | 顺位 8 | 实现本体 = 顺位 8。 |

**§4.2.7 / §E2 写入的绑定门（非软提示）**：顺位 8 实施 throwing decoder 时**必须**同 PR 执行 CONTRACT_VERSION/m01/acceptance 三件套 reconciliation；spec 在此 codify 该义务为 MANDATORY，顺位 8 评审须核此门。
