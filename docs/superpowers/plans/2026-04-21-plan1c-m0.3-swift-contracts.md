> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 冻结 M0.3 Swift 数据模型契约——新建 `ios/Contracts/` SwiftPM 独立包，交付 3 个 Swift 源文件（Models.swift / AppState.swift / RESTDTOs.swift）+ 对应单元测试 + Package.swift + fixture JSON + CI workflow + 聚合验收。所有值类型 `Sendable`（spec v1.3 要求）；Codable 字段与 M0.1 DDL / M0.2 OpenAPI 的 snake_case 命名完全对齐。

**Architecture:**
- **独立 SwiftPM 包**：`ios/Contracts/` 与现有 Xcode 工程（`ios/KlineTrainer.xcodeproj`）解耦；`swift test` 从 CLI 直接跑，零 Xcode GUI 依赖（memory `feedback_xcode_cli_first.md`）。Plan 2/3 Xcode 工程把本包作为 local package 挂进来消费。
- **两层验证延续 Plan 1/1b pattern**：
  - 本地：`swift test`（Swift Testing 框架，Swift 6.0+ 原生）
  - CI：`.github/workflows/swift-contracts-smoke.yml`，macos-15 runner + 内置 Swift 6.0，再跑一遍 `swift test`
- **契约跨文件一致性**（Plan 1/1b 已建立，Plan 1c 对齐）：
  - `CONTRACT_VERSION = "1.4"` 常量声明在 Models.swift 顶层（与 docs/contracts/contract-version-matrix.md 对齐）
  - `TrainingSetMetaItem.contentHash` 字段验证在测试里加 `^[0-9a-f]{8}$` pattern 断言（与 Plan 1 `ck_content_hash_crc32_lowercase` + Plan 1b OpenAPI `pattern` 对齐）
  - `LeaseResponse` snake_case CodingKeys 与 Plan 1b OpenAPI 字段名一致（`lease_id` / `expires_at`）
  - `TrainingSetMetaItem` 7 字段与 Plan 1b OpenAPI schema 7 字段对齐

**Tech Stack:** Swift 6.0+ / Swift Testing / Swift Package Manager / GitHub Actions macos-15 runner。

**依赖（hard prereq）**：
- Plan 1（M0.1 DDL）已 merged（PR #23）—— 本 plan 对齐 `content_hash` format
- **Plan 1b（M0.2 OpenAPI）必须 merge 后再进入 Plan 1c 实施阶段**（codex round 1 finding：acceptance 验收项 4 grep 三文件含 `backend/openapi.yaml`，若 Plan 1b 未 merge 本地 acceptance 必然 FAIL）。plan 文档可先起草，但 Task 1-5 实施前必须 git merge origin/main 拉进 Plan 1b artifacts

**Scope 说明**：
- 本 plan 只冻结 **Swift 数据模型**（struct/enum + Equatable/Sendable；**大多数** 类型额外含 Codable + CodingKeys）
- **Codable 例外（spec M0.3 故意设计，codex round 5 澄清）**：
  - `TrainingSetFile: Equatable, Sendable` —— **无 Codable**。理由：runtime 缓存元数据（字段含 `localURL: URL` 文件系统引用），由 P5 CacheManager 在 app 运行时管理；不跨进程/跨版本持久化；spec M0.3 逐字定义未声明 Codable。
  - `AppSettings: Equatable, Sendable` —— **无 Codable**。理由：以 key-value 形式存 app.sqlite `settings` 表（每字段一行 TEXT，不整体 JSON-encode），P6 SettingsStore 在 get/set 时按字段映射；spec M0.3 逐字定义未声明 Codable。
  - 其它**所有**值类型（enums / KLineCandle / TrainingSetMeta / FeeSnapshot / TradeOperation / Drawing* / TrainingRecord / DrawdownAccumulator / PendingTraining / LeaseResponse / TrainingSetMetaItem）**都是** Codable + Equatable + Sendable。
- **不包含**：M0.4 AppError（→ Plan 1d）、M0.5 并发规则文档（→ Plan 1e）、Plan 3 P1 APIClient 实现（→ Plan 3）
- **不 Xcode 工程集成**：SwiftPM 包独立存在，Xcode 工程 Plan 2/3 再通过 "Add Local Package" 一次性 GUI 集成

**Codable key 命名策略说明（codex round 1 finding #2 回应）**：

spec M0.3 为 `KLineCandle` 显式给出 snake_case `CodingKeys`（`boll_upper` 等），但 `TradeOperation` / `TrainingRecord` / `PendingTraining` / `DrawingObject` / `FeeSnapshot` 等**不含**显式 CodingKeys（comment "Codable 保持 rawValue ... 无需自定义 CodingKeys" 是针对 PositionTier enum，不是 TradeOperation 字段层）。本 plan 忠实 spec，**不添加** CodingKeys 到这些结构体。

**snake_case DB 列和 camelCase Swift 属性的映射归 GRDB DAO 层**（Plan 3 P4 落地）：

- Plan 3 P4 AppDB 的 `DatabaseMigrator` 配置必须设 `columnDecodingStrategy = .convertFromSnakeCase` 和 `columnEncodingStrategy = .convertToSnakeCase`（GRDB 6.x+ API）
- 在这个 strategy 下：`positionTier ↔ position_tier` / `globalTick ↔ global_tick` / `createdAt ↔ created_at` 等**按规则自动转换**，无需 struct 层 CodingKeys
- **KLineCandle 例外**：显式 CodingKeys 保留因为 training-set SQLite（P3a TrainingSetReader）独立于 P4 AppDB 使用 GRDB，且 training-set 里的 `boll_upper` / `macd_diff` 等命名符合相同规则，CodingKeys 是 spec 的防御性明确写法（belt-and-suspenders）
- **RESTDTOs 例外**：`LeaseResponse` / `TrainingSetMetaItem` 有显式 CodingKeys，因为 REST JSON 解码**不**走 GRDB；依赖 Swift 标准 `JSONDecoder`，必须 struct 层 CodingKeys 桥接 snake_case JSON 字段名

**本 plan 需要做的跨层契约确保**：
1. 在 Task 2 AppState.swift 顶部加一段 doc comment 明确 "snake_case ↔ camelCase 由 P4 DAO columnDecodingStrategy 配置"，便于 Plan 3 P4 owner 看到期望
2. 在 Task 2 AppStateTests.swift 加 1 个断言 "TradeOperation JSON 默认编码是 camelCase（_not_ snake_case）"，证明模型层确实是 camelCase，snake_case 转换归 GRDB 侧
3. Plan 3 P4 DAO 写 CodingKeys 映射**不是** Plan 1c scope——但若 Plan 3 P4 实施时发现 `convertFromSnakeCase` 策略不够（比如有字段需要特殊映射），回来 bump Plan 1c 加显式 CodingKeys

        case bollLower = "boll_lower"
        case macdDiff = "macd_diff"
        case macdDea = "macd_dea"
        case macdBar = "macd_bar"
        case globalIndex = "global_index"
        case endGlobalIndex = "end_global_index"
    }

public struct TrainingSetMeta: Codable, Equatable, Sendable {
    public let stockCode: String
    public let stockName: String
    public let startDatetime: Int64
    public let endDatetime: Int64

    public init(stockCode: String, stockName: String, startDatetime: Int64, endDatetime: Int64) {
        self.stockCode = stockCode
        self.stockName = stockName
        self.startDatetime = startDatetime
        self.endDatetime = endDatetime
    }

    // 显式 CodingKeys（codex round 4 finding）：TrainingSetMeta 从 training-set

        case startDatetime = "start_datetime"

    }
}

// MARK: - Drawdown Accumulator（v1.3 新增）

public struct DrawdownAccumulator: Codable, Equatable, Sendable {
    public var peakCapital: Double
    public var maxDrawdown: Double

    public init(peakCapital: Double, maxDrawdown: Double) {
        self.peakCapital = peakCapital
        self.maxDrawdown = maxDrawdown
    }

    /// 每 tick 或每次交易后调用。peakCapital 单调上升；maxDrawdown 单调上升（候选 = peak - current）。
    public mutating func update(currentCapital: Double) {
        if currentCapital > peakCapital { peakCapital = currentCapital }
        let dd = peakCapital - currentCapital
        if dd > maxDrawdown { maxDrawdown = dd }
    }

    public static let initial = DrawdownAccumulator(peakCapital: 0, maxDrawdown: 0)
